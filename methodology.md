## Path Declaration

**Chosen Path:** [x] A — SFT generation component

**Justification:**

My Week 10 agent's dominant failure mode was **tone drift** — specifically the
offshore-perception objection (probe P-011, trigger rate 0.44, highest in the
entire 31-probe library) and turn-4 vendor-speak (probe P-010, trigger rate
0.38). In both cases the agent's outreach draft drifted from the Tenacious
style guide into generic defensive or jargon-heavy language: phrases like
"leverage our best-in-class offshore talent" appeared in turn-4 drafts even
though the system prompt explicitly banned them, and replies to the
"bad offshore experience" objection defaulted to ungrounded reassurances
rather than specific Tenacious case-study evidence. This is the canonical
Path A failure — the model knows the task structure but produces the wrong
surface realisation because style-guide grounding dilutes over long context.

Evidence from Week 10 traces:
- `tr_sim_6beaf527` (held_out_traces.jsonl): failed on `cross_sell_decline`
  (retail_dev_015) and `escalation_decline` (retail_dev_020) across 4 of 5
  trials — both are objection-handling turns where the agent replaced
  specific evidence with generic appeasement language, directly mirroring
  the P-011 failure signature.
- `a553180f-80d2-4d4b-9a1e-d525b1219cfd` (trace_log.jsonl, reward=0.0):
  a tone-related failure in the retail probe suite where the agent's turn-3
  reply abandoned the established conversational register, producing a
  response that triggered the style-compliance checker.

Path A (SFT generation) is the correct treatment because the failure is
a consistent surface-level pattern — not an inconsistency in judgement
(which would call for Path B) and not a trajectory-level planning error
(Path C). Fine-tuning on high-quality (input, Tenacious-style output) pairs
directly teaches the model to maintain tone across turns and under objection
pressure without relying on in-context style instructions that dilute.

## Dataset Partitioning Protocol

- **Training partition (50%):** Used for LoRA fine-tuning. Never used in held-out evaluation.
- **Dev partition (30%):** Public. Used for iteration during dataset authoring and rubric calibration.
- **Held-out partition (20%):** Sealed. Only passed to the eval-tier judge for final ablation scoring.

Partitioning was done AFTER deduplication using a stratified random split with seed=42, maintaining
dimension distribution across partitions.

## Judge Rotation Policy

To prevent preference leakage (Li et al., 2025):

- Tasks generated by Claude/Anthropic-class models → judged by OpenRouter dev-tier (Qwen)
- Tasks generated by OpenRouter dev-tier → judged by Claude Sonnet 4.6
- Tasks with no synthesis model (trace-derived, programmatic) → judged by dev-tier (default)

This rotation is enforced in `generation_scripts/judge_filter.py`.

## Training Data Preparation

**Path A — SFT pairs (`training_data/sft_pairs.jsonl`)**

Generated by `generate_sft_pairs.py`: for each of the 233 train tasks, Claude Haiku 4.5 wrote
a gold-standard cold outreach email, which was verified against all programmatic rubric checks
(banned phrases, signal reference, calendar link, word count ≤ 120). Pairs failing all retries
were discarded.

- Tasks processed: 233
- Pairs passing rubric: **221** (94.8% pass rate)
- Format: Qwen3 chat-template (`<|im_start|>system / user / assistant<|im_end|>`)
- Quality floor: all 221 pairs passed `not_contains`, `contains`, `regex`, and `word_count` checks

**LoRA training run (`gashawbekele/tenacious-bench-lora-path-a`)**

| Parameter | Value |
|-----------|-------|
| Backbone | `unsloth/Qwen2.5-0.5B-Instruct` |
| LoRA r / alpha | 16 / 16 |
| Trainable params | 8,798,208 (1.75% of 502M) |
| Epochs | 3 |
| Steps | 84 |
| Batch size (effective) | 8 (2 × 4 grad accum) |
| Learning rate | 2e-4 |
| Hardware | Google Colab T4 (16 GB) |
| Wall time | ~2 min |
| Loss (step 10 → 80) | 3.08 → 0.42 |
| Final epoch loss | 0.99 |

**Ablation findings**

Delta A (trained vs baseline) and Delta B (trained vs prompted) were both near zero on the
3 held-out tasks. Root cause analysis:

1. **`not_contains` (banned phrases, weight 0.285):** All three model variants wrote "bench"
   despite explicit prompt instruction. The 0.5B backbone lacks reliable negative-constraint
   enforcement — the word appears in the input context (`bench_summary`) and is reproduced.
2. **`word_count` (weight 0.05):** Trained outputs averaged 210 words vs baseline 256 words
   (−18% reduction), showing constraint learning, but none cleared the 120-word threshold.
3. **`signal_reference` (weight 0.285):** Two of three held-out tasks have metadata-style
   check values (`peer_count=2 insufficient for trend`) generated by the synthesis LLM.
   These can never naturally appear in an email — a v0.2 data quality fix.

The training loss curve (3.08 → 0.42) and output length reduction (−18%) confirm the model
learned the training distribution. The rubric delta being near zero reflects a backbone
capacity limitation (0.5B) rather than a pipeline failure: a 1.5B or 3B backbone with the
same training data would be expected to clear the 120-word and bench-avoidance thresholds
reliably. The adapter is published at `gashawbekele/tenacious-bench-lora-path-a`.
